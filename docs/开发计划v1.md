## 二、核心原则

| 原则 | 说明 |
|------|------|
| **依赖阻塞** | 被依赖的模块先开发（Database → P1/P2 → P3） |
| **契约驱动** | API 文档已定，前端可以用 mock 数据并行开发 |
| **独立可测** | 每个模块能单独验证，不依赖其他模块 |
| **并行最大化** | P1、P2、P4 无依赖关系，同时开发 |

## 三、详细任务拆分

### 阶段 0：基础设施搭建（P3 主导，第 1 周）

> **目标：** 搭建后端编译环境、数据库层、前端脚手架，为其他模块提供开发基础

| 任务 | 说明 | 产出 |
|------|------|------|
| 0.1 CMake 配置 | 配置 vcpkg，添加第三方库依赖（miniaudio, TagLib, SQLite, cpp-httplib, nlohmann/json） | 可编译的 CMakeLists.txt |
| 0.2 数据库模块 | Database 类：连接管理、建表、DAO 接口 | database/ 下可独立测试 |
| 0.3 数据模型 | SongInfo、PlayerState、PlayMode 结构体定义 | models/ 下的头文件 |
| 0.4 前端脚手架 | Vue3 + Vite + Element Plus + Pinia + vue-router 初始化 | 可运行的前端空项目 |

**验收标准：**
- C++ 后端能编译通过
- `data/59music.db` 能自动创建，表结构正确
- 前端 `npm run dev` 能启动

**为什么必须先行：**
- CMake 是 C++ 编译的前提，没有它所有后端代码无法运行
- Database 是 P1（播放队列持久化）、P2（歌曲入库）、P3（API 查询）的共同依赖
- 前端脚手架是 P4 开发的前提

---

### 阶段 1：核心模块并行开发（P1 + P2 + P4，第 1-2 周）

> **目标：** P1、P2、P4 三个模块完全并行开发，各自独立可测试

#### P1：播放引擎模块（独立开发）

| 任务 | 说明 | 依赖 |
|------|------|------|
| 1.1.1 音频解码器 | AudioDecoder：封装 miniaudio，open/decode/seek/close | 无 |
| 1.1.2 PlayerEngine | 播放控制：play/pause/stop/next/prev/seek/volume | 1.1.1 |
| 1.1.3 播放队列 | PlayQueue：维护歌曲列表、当前索引、播放模式切换 | 无 |
| 1.1.4 播放模式 | 实现顺序/单曲循环/列表循环/随机四种模式 | 1.1.3 |

**验收标准：**
- 命令行能播放一个本地音频文件
- 暂停/恢复/跳转进度正常
- 切换播放模式后行为正确

**独立开发理由：**
- 不依赖 Database（播放逻辑本身不需要数据库）
- 不依赖 P2（播放不关心歌曲怎么扫描入库）
- 可以用硬编码的文件路径测试

---

#### P2：音乐库模块（独立开发）

| 任务 | 说明 | 依赖 |
|------|------|------|
| 1.2.1 文件扫描器 | FileScanner：递归扫描文件夹，过滤 .mp3/.flac/.wav | 无 |
| 1.2.2 元数据解析 | MetadataParser：TagLib 读取歌名/歌手/专辑/时长 | 无 |
| 1.2.3 LibraryManager | 整合扫描+解析+入库，提供查询/删除接口 | 0.2 Database |

**验收标准：**
- 命令行调用扫描文件夹，数据库中能查到歌曲记录
- 分页查询、关键词搜索正常

**独立开发理由：**
- 依赖 Database，但阶段 0 已完成
- 不依赖 P1（入库不需要播放能力）
- 可以独立测试扫描和查询

---

#### P4：前端开发（独立开发，用 mock 数据）

| 任务 | 说明 | 依赖 |
|------|------|------|
| 1.4.1 布局框架 | 主布局：侧边栏 + 主内容区 + 底部播放栏 | 0.4 脚手架 |
| 1.4.2 API 封装 | axios 封装所有后端接口（先用 mock 数据） | API 文档 |
| 1.4.3 音乐库页面 | 歌曲列表、搜索、扫描文件夹 | 1.4.2 |
| 1.4.4 播放队列页面 | 队列展示、拖拽排序、清空 | 1.4.2 |
| 1.4.5 播放控制栏 | 播放/暂停、进度条、音量、播放模式 | 1.4.2 |
| 1.4.6 状态管理 | Pinia store 管理播放状态和队列 | 1.4.2 |

**验收标准：**
- 所有页面正常展示
- 用 mock 数据能模拟播放流程
- UI 交互逻辑正确

**独立开发理由：**
- API 文档已完成，接口契约明确
- 前端可以用 mock 数据开发 UI，不依赖后端实现
- 后续对接时只需替换 mock 为真实请求

---

### 阶段 2：服务层整合（P3 主导，第 3 周）

> **目标：** 整合 P1、P2 模块，实现全部 HTTP API 和 WebSocket 推送

| 任务 | 说明 | 依赖 |
|------|------|------|
| 2.1 HttpServer 骨架 | cpp-httplib 搭建服务，注册路由，JSON 响应封装 | 0.1 |
| 2.2 播放器 API | 实现 /player/* 8 个接口 | 1.1.x + 2.1 |
| 2.3 队列 API | 实现 /queue/* 5 个接口 | 1.1.3 + 2.1 |
| 2.4 音乐库 API | 实现 /library/* 4 个接口 | 1.2.x + 2.1 |
| 2.5 WebSocket 服务 | 推送 player_state / progress / queue_changed / library_updated | 2.1 |
| 2.6 main.cpp | 整合所有模块，启动服务 | 全部 |

**验收标准：**
- curl 能调通所有 REST API
- WebSocket 客户端能收到状态推送
- 重启后播放队列和设置能恢复

**为什么放在阶段 2：**
- P3 是"胶水层"，不实现业务逻辑，只负责 HTTP 路由 → 调用 P1/P2 → 返回 JSON
- 必须等 P1、P2 接口稳定后才能对接
- P3 可以在阶段 0 就搭好 HttpServer 骨架，但完整对接需要等阶段 1 完成

---

### 阶段 3：前端对接 + 联调测试（全员，第 4 周）

> **目标：** 前端对接真实 API，端到端功能验证

| 任务 | 说明 | 负责 |
|------|------|------|
| 3.1 前端对接 | 替换 mock 数据为真实 API 请求 | P4 |
| 3.2 WebSocket 对接 | 实时接收播放状态和进度更新 | P4 |
| 3.3 端到端测试 | 扫描 → 搜索 → 加入队列 → 播放 → 切歌 → 完整流程 | 全员 |
| 3.4 异常处理 | 文件不存在、格式不支持、网络断开等边界情况 | 全员 |
| 3.5 体验优化 | 加载状态、错误提示、操作反馈 | 全员 |

**为什么联调是最后一步：**
- 联调的前提是所有模块都可用
- 联调时才会发现接口不一致、数据格式问题、边界情况

## 四、并行开发时间线

```
第 1 周
├── P3：阶段 0（CMake + Database + 数据模型）        ← 基础设施
├── P1：1.1.1 AudioDecoder + 1.1.2 PlayerEngine     ← 独立开发
├── P2：1.2.1 FileScanner + 1.2.2 MetadataParser    ← 独立开发
└── P4：阶段 0 前端脚手架 + 1.4.1 布局框架           ← 独立开发

第 2 周
├── P1：1.1.3 PlayQueue + 1.1.4 播放模式            ← 独立开发
├── P2：1.2.3 LibraryManager（接入 Database）        ← 依赖阶段 0 完成
├── P3：2.1 HttpServer 骨架                         ← 搭框架
└── P4：1.4.2 ~ 1.4.6 全部页面 + 状态管理           ← 用 mock 数据

第 3 周
├── P3：2.2 ~ 2.6 整合 P1/P2，实现全部 API          ← 依赖阶段 1 完成
├── P4：3.2 WebSocket 对接                          ← 独立开发
└── P1/P2：协助 P3 对接，修复模块问题               ← 支持角色

第 4 周
├── P4：3.1 前端对接真实 API                         ← 依赖阶段 2 完成
└── 全员：3.3 ~ 3.5 联调测试、Bug 修复、体验优化     ← 最终阶段
```

## 六、风险与应对

| 风险 | 应对 |
|------|------|
| miniaudio 编译问题 | 先用最简单的 wav 测试，再扩展 mp3/flac |
| TagLib 依赖复杂 | 使用 vcpkg 安装，避免手动编译 |
| WebSocket 状态同步 | 前端断线重连机制 |
| 播放队列并发 | 后端单线程处理，避免锁竞争 |
| 前后端接口不一致 | 严格遵循 API 文档，联调时及时沟通 |
